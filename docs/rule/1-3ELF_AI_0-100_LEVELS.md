# 1-3ELF_AI_0-100_LEVELS - AI 理解階段與行為總表

> 本文件搭配 `1-2ELF_AI_ONBOARDING` 使用，用來快速查看：
>
> - 0–50 / 50–70 / 70–90 / 90–100 四個階段各自是什麼狀態
> - 每一階段 AI 應該「做什麼」與「產出哪些知識文件」
> - 每一階段「不應該做什麼」

---

## 概覽

- **階段一：0–50 分**探索期（只許理解與紀錄，不許下大結論與大改動）
- **階段二：50–70 分**模組成形（重要模組已清楚，可開始寫模組級總覽）
- **階段三：70–90 分**整體成形（全系統關係圖成形，可開始做整合與架構級規劃）
- **階段四：90–100 分** 成熟維運（以維護現有總覽、協助他人 onboarding 為主）

> 誠信與收斂原則，詳見：`1-2ELF_AI_ONBOARDING` 第 0 節。

---

## 階段一：0–50 分（探索期）

**定位**

- 剛進專案，只做初步掃描。
- 對 domain / 技術棧只有「第一印象」，隨時可能被推翻。

**主要目標**

- 知道「這大概是什麼系統」、檔案大概怎麼分布。
- 為後續深入閱讀列出計畫。

**AI 應該做的事**

- 閱讀：
    - README / LICENSE / 任何現有架構說明。
    - 專案根目錄與主要子資料夾（只看結構，不深挖內容）。
- 紀錄：
    - 產出至少一份「掃描紀錄」，內容包含：
        - 已讀檔案與簡短印象。
        - 初步猜測專案用途／domain（標註為「推測」）。
        - 明確列出尚未閱讀的區塊與後續閱讀計畫。

**這一階段「必須」產出的知識**

- 1 份以上「掃描紀錄」（可以是 note / TODO / 任意檔案，但需存在 repo 中）：
    - 已讀／未讀檔案清單。
    - 第一版專案用途與技術棧的猜測。
    - 下一步要看的檔案列表。

**這一階段不應該做的事**

- 不宣稱自己已經理解整體架構。
- 不對部署、安全、資料庫 schema 提出大膽建議。
- 不進行任何結構性修改（refactor、Docker 大改、DB 變更）。

---

## 階段二：50–70 分（模組成形）

**定位**

- 已知道專案大概在做什麼。
- 前端 / 後端 / DB / Infra 等主要模組已能分辨。
- 能說出「幾個最重要的功能區塊」。

**主要目標**

- 讓任何人一打開幾份關鍵文件，就能理解「專案長什麼樣」。
- 把「模組級」的理解固定下來，成為穩定的進入點。

**AI 應該做的事**

- 針對整體與主要模組產出：
    - `PROJECT_OVERVIEW` 初版：
        - 專案用途 / domain。
        - 技術棧（前端 / 後端 / DB / Infra）概觀。
        - 主要資料夾職責、啟動 / build / 測試指令。
    - `FRONTEND_ARCH` 初版（如有前端）：
        - 前端啟動流程。
        - 路由結構與主要區塊。
        - 狀態管理與 UI / Design System 概観。
    - `API_OVERVIEW` 初版（視專案規模）：
        - 以 domain 或模組分區列出 API 區塊與職責。
        - 指向正式 API 文件（Swagger / OpenAPI 等）。
- 視需要，為 1–3 個關鍵功能撰寫 `ui-flow-<feature>.md`：
    - 描述使用者操作路徑。
    - 描述前端 / API / store / DB 的資料流向。

**這一階段「必須」產出的知識**

- `PROJECT_OVERVIEW` 初版。
- 若有前端：`FRONTEND_ARCH` 初版。
- 若後端複雜：`API_OVERVIEW` 初版。
- 若有非常核心功能：至少 1–3 份 `UI-FLOW-<FEATURE>.md`。

**這一階段不應該做的事**

- 不為同一主題開多個 overview 檔（例如 `PROJECT_OVERVIEW_v2.md`）；應在原檔內迭代。
- 不宣稱「全專案技術棧已定案」，因為跨模組依賴仍可能沒看完。

---

## 階段三：70–90 分（整體成形）

**定位**

- 大部分關鍵模組都看過，也看過它們彼此怎麼互動。
- 能畫出一張合理的系統架構圖。

**主要目標**

- 把前面零散的理解，整合成「全域視角」：
    - 技術棧總表。
    - 系統架構圖。
    - 部署路線圖（必要時）。

**AI 應該做的事**

- 產出並維護：
    - `TECH_STACK_OVERVIEW`：
        - 前端 / 後端 / DB / Cache / Queue / Infra / DevOps 工具列表。
        - 各技術在專案中的角色與邊界。
    - `ARCH_OVERVIEW`：
        - 用文字 + 圖（Mermaid 等）描述：
            - Browser / Client → Web → API → DB / Redis / Queue / 其他服務。
            - 重要資料流與外部依賴。
    - `DEPLOYMENT_OVERVIEW`（必要時）：
        - 開發 / 測試 / 正式環境的部署方式。
        - 主要指令、環境變數、注意事項。
- 整合早期筆記：
    - 將仍有效的資訊搬入上述總覽檔。
    - 幫過時或重複內容加註記（例如「已過時，請看 XXX 檔」）。

**這一階段「必須」產出的知識**

- `TECH_STACK_OVERVIEW`。
- `ARCH_OVERVIEW`（含圖）。
- 視需求補完或新增 `DEPLOYMENT_OVERVIEW`。

**這一階段不應該做的事**

- 不再隨意增加新的 Overview 類檔案，而是整合到現有總覽。
- 不在總覽檔堆疊一次性的 debug 細節（應留在工作日誌 / TODO）。

## 階段四：90–100 分（成熟維運）

**定位**

- 已能預測「改任意一塊，大致會影響哪裡」。
- 適合參與架構決策、規劃大型重構或新功能。

**主要目標**

- 維護文件的一致性與更新度。
- 協助新成員與新 AI 快速上手。

**AI 應該做的事**

- 針對每次重要變更：
    - 更新 `PROJECT_OVERVIEW` / `FRONTEND_ARCH` / `API_OVERVIEW` / `TECH_STACK_OVERVIEW` /
      `ARCH_OVERVIEW` / `DEPLOYMENT_OVERVIEW` 中相關段落。
    - 說明這次變更對架構與技術棧的影響。
- 協助 Onboarding：
    - 告訴新成員 / 新 AI「先讀哪幾份檔案」。
    - 根據實際情況調整文件順序與內容，避免舊資訊誤導。

**這一階段不應該做的事**

- 不再胡亂新增新的 Overview 型檔案。
- 不把短期試驗想法直接寫進核心規則或總覽檔，避免混淆「長期真相」與「短期假設」。
