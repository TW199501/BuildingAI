# TODO 2025-11-23 01

## 主題：第二階段：Docker 升級與啟動體驗優化

### [問題標題]

- 目前使用的 Docker 部署方式雖然可以成功啟動服務，但存在以下痛點：
    - 第一次拉取鏡像與安裝依賴耗時較長（包含從遠端 registry 拉取 Redis/Postgres/Node 基底鏡像，以及在 node 容器內執行
      `pnpm install`）。
    - 每次重新啟動 node 容器時，仍會在容器內執行依賴檢查與安裝流程，啟動體驗不夠「即起即用」。
- 目標是在不破壞現有功能的前提下，分階段優化 Docker 方案，讓本地與正式環境的啟動速度與可預期性都更好。

### [背景與重現步驟]

- 目前的啟動流程（已在 TODO 2025-11-22-09 中驗證）：
    - `pnpm build` 在本機可成功執行，所有 Turbo 任務完成（20/20 成功）。
    - `pnpm docker:up` 可順利啟動三個主要容器：
        - `buildingai-redis`（使用 `ccr.ccs.tencentyun.com/buildingai/redis:8.2.2`）。
        - `buildingai-postgres`（使用 `ccr.ccs.tencentyun.com/buildingai/postgres:17.6`）。
        - `buildingai-nodejs`（使用 `ccr.ccs.tencentyun.com/buildingai/node:22.20.0`）。
    - 初次啟動時，Node 容器 log 中可見大量
      `Progress: resolved 2198, reused ..., downloaded ..., added ...`，代表容器內正在執行
      `pnpm install`，耗時可能接近 1 小時。
- 痛點總結：
    - **鏡像來源**：Redis/Postgres/Node 等基底鏡像目前從大陸 registry 拉取，在台灣/其他區域可能較慢。
    - **安裝時機**：依賴安裝發生在「容器啟動時」（entrypoint 中），而不是 build
      image 階段，導致每次啟動都要「等容器慢慢跑」。

### [相關檔案路徑與定位]

- Docker / Compose：
    - `docker-compose.yml`
    - （未來可能新增）`Dockerfile`、`docker-compose.override.yml` 或其他環境區分用的 compose 檔。
- 應用入口與 scripts：
    - 根目錄 `package.json` 的 `start` / `build` / `docker:*` scripts。
    - `packages/api`、`packages/web/buildingai-ui`、`packages/@buildingai/*` 等在 `pnpm build`
      中參與的套件。
- 既有說明文件：
    - `README.md` / `README.zh-CN.md`（目前以 Docker 為主要安裝方式）。
    - `docs/todo/todo2025-11-22-09.md`（第一階段 build / Docker 驗證記錄）。

### [分析與方案]

- 分析：
    - 現有 Docker 方案偏向「快速在任何地方跑起 repo」：
        - 使用 Node 基底鏡像 + bind mount 原始碼 + 容器內 `pnpm install` / `pnpm start`。
        - 優點：開發時方便直接掛本機程式碼；缺點：啟動時間長且每次啟動都需要檢查依賴。
    - 資料庫與 Redis 採用遠端 registry 的 image，在非大陸網路環境拉取速度不穩定。
    - 對於「正式部署」或「長期運行」場景，更建議：
        - 將 `pnpm install` / `pnpm build` 放在 Docker build 階段完成，產出可重複部署的應用 image。
        - 使用更靠近實際部署環境的官方 Redis/Postgres
          image（如無特別需求），提升下載速度與可預期性。

- 方案（第二階段分步計畫，僅設計，不馬上一次全部修改）：
    1. **確認與記錄現有 Docker 行為（基準線）**
        - 目標：明確記錄當前方案的啟動時間與關鍵行為，作為後續優化的對照組。
        - 操作：
            - 重新執行一次 `pnpm docker:up`，記錄：
                - 從執行指令到 Redis/Postgres/Node 三個容器 `Up (healthy)` 的時間。
                - Node 容器從 `health: starting` 轉為可正常回應 `/consoleapi/health` 的時間。
            - 在本檔「實作重點摘要」中補上實際耗時與觀察到的 log 關鍵片段。

    2. **設計 Node 應用 multi-stage Dockerfile（應用 image 化）**
        - 目標：把 `pnpm install` / `pnpm build`
          前移到 build 階段，產出一個可直接啟動的 BuildingAI 應用 image。
        - 規劃草案：
            - `Dockerfile`（初版構想）：
                - `FROM node:22` 作為 builder stage：
                    - `WORKDIR /app`
                    - `COPY . .`
                    - 啟用 corepack / pnpm，執行 `pnpm install --frozen-lockfile`。
                    - 執行 `pnpm build`，產生 `.output` / `dist` 等產物。
                - `FROM node:22-slim` 作為 runtime stage：
                    - `WORKDIR /app`
                    - 只 `COPY` builder stage 中的最小必要產物（例如 `.output`、必要的 config
                      / 靜態檔）。
                    - 設定 `NODE_ENV=production`，暴露 4090 port，`CMD` or `ENTRYPOINT`
                      執行對應的 start script。
            - 此階段僅在本 TODO 中設計與列出需求，實作細節與實際路徑（例如 `.output` vs
              `dist`）將在後續子任務中透過閱讀 `packages/api` / `packages/web/buildingai-ui`
              的 build 結果進一步確認。

    3. **規劃 docker-compose 與 Dockerfile 的分環境策略**
        - 目標：同時兼顧「開發方便」與「正式部署穩定」。
        - 構想：
            - 保留現有 `docker-compose.yml` 作為「開發 / 快速體驗」路線：
                - 使用 bind mount 掛載原始碼，容器內跑 `pnpm start`，適合需要即時修改程式碼的場合。
            - 新增例如 `docker-compose.prod.yml` 或 `docker-compose.override.yml`：
                - Node 服務改為使用本地 build 的應用 image（`build: .` 或
                  `image: my-buildingai-app:tag`）。
                - 不再在容器內掛整個 repo，也不在啟動時計算 `pnpm install`。
        - 此步驟需要先完成 Dockerfile 設計，並確認正式部署時需要的環境變數與 volume 掛載策略。

    4. **評估與優化 Redis / Postgres 鏡像來源**
        - 目標：減少從遠端 registry 拉取基底鏡像的時間，提升首次啟動體驗。
        - 思路：
            - 分析目前使用的 `ccr.ccs.tencentyun.com/buildingai/redis:8.2.2` / `.../postgres:17.6`
              是否有客製化設定（如時區、預設配置）。
            - 若與官方 `redis:8.2.2` / `postgres:17.6` 差異不大，可考慮：
                - 在新 compose 檔中（例如 `docker-compose.local.yml`）使用 Docker
                  Hub 官方 image，專供本機開發使用。
            - 若官方 image 行為與現有 image 差異較大，則需在本檔記錄差異點，評估是否要：
                - 維持現有 image 作為正式部署版本。
                - 另行建立自家 registry / mirror 以改善下載速度。

    5. **更新文件與使用說明**
        - 目標：讓未來開發者清楚區分「快速體驗」與「正式部署」兩種 Docker 路線。
        - 操作：
            - 在 `README.md` / `README.zh-CN.md` 中新增小節：
                - 說明：
                    - 基本 `docker compose up -d`（快速體驗，可能第一次較久）。
                    - 進階部署：使用 multi-stage Dockerfile + 專用 compose 檔的方案。
            - 在 `docs/` 新增（或擴充）部署相關說明文件，包含：
                - 映像版本命名建議。
                - 所需環境變數清單與 `.env` 模板。
