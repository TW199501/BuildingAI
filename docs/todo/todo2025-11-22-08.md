## 2. 我目前對這個專案的理解度到哪裡？

### 2.1 全局與前端

我現在對以下部分有**相對完整的心智模型**：

- **整體架構**

  - monorepo 結構（`packages/api`、`packages/web/buildingai-ui`、`packages/web/@buildingai/**`、stores、service 層）。
  - 前端技術棧：Nuxt 4 + Vue 3 + Pinia + Nuxt UI + TailwindCSS。
  - 後端：NestJS + TypeORM + PostgreSQL（從 system/chat/auth 模組閱讀來的）。
- **前端框架層**

  - `nuxt.config.ts`（project 與 shared `@buildingai/nuxt`）、`app/app.vue` 啟動流程。
  - `app/middleware/route.global.ts` 的初始化檢查、登入導向、權限與動態 console 路由註冊。
  - console 佈局（`app/layouts/console.vue`）與 `app` / `user` stores 的職責。

### 2.2 Feature 級前後端流程

你指定的 A/B/C/D 部分，我的理解度是「可以講出端到端資料流」的程度：

- **Feature A：install**

  - 前端：`/install` 四步驟 UI、`useInstall`、`apiInitializeSystem` 呼叫與成功後 user/app store 更新。
  - 後端：System 模組 `initialize` 的 controller / DTO / service，DB 寫入哪些表（user、dict webinfo、SYSTEM_CONFIG.isInitialized）、如何產生 token。
- **Feature B：login/auth**

  - 前端：`/login` index + account / wechat / login-bind 三子流程、`userStore` 狀態與 `handleAuth()`。
  - 後端：`/auth/login`、wechat qrcode／ticket、`/user/info`、SMS 傳送與 `bindMobile` 大致用途（細節保留給未來安全深挖）。
- **Feature C：chat**

  - 前端：`/chat/:id` 主頁、conversation list / messages / prompt 元件、`useChat` composable、SSE 事件型別與狀態生命週期。
  - 後端：`/ai-chat` + `/ai-chat/stream`、`/ai-conversations/**`，以及一整組 handlers（model validation、user power、MCP init & tool calls、message context trimming、title generation、power deduction）。
    資料流、錯誤處理、token→power 的轉換邏輯都有整理。
- **Feature D1：console – AI Provider 管理（前端）**

  - 前端：`console/ai/provider/list.vue` + `edit.vue` 的 UI flow：
    - 列表搜尋／篩選／批次刪除／啟用切換。
    - modal 表單欄位、yup 驗證、KeyPoolSelect、前往 secret 管理的跳轉。
  - 後端：目前只從前端推演出「有一組 consoleapi 的 provider 相關路由」，但**尚未展開後端 provider/secret/AiModel 模組**。

## 3. 就這樣的理解度，能不能做「全面性修改」？

我會分兩層回答：**可以放心做的範圍**，以及**暫時不要一口氣大改的範圍**。

### 3.1 相對可以放心做較大修改（仍會小步、文件優先）

在下列區塊，如果你要做「結構性調整」或「新功能」，我有足夠上下文可以一起設計＋實作（前提仍是按 ELF EXPRESS：先文件、再小步 code）：

- **前端**

  - install / login / chat 的主要頁面、組件、composable、stores。
  - console AI Provider 前端頁面的 UX / 資料流調整。
  - 全域路由 middleware 與 console 佈局，若改動是延伸既有邏輯（例如新增某種 auth 條件，而不是完全重寫）。
- **後端（局部而不是全域）**

  - **install 初始化 API**：增減欄位、更改初始化策略（例如多一個站點設定），因為 call stack 與 DB 寫入流程已掌握。
  - **login/auth 主要路由**：登入行為、使用者資訊回傳欄位的調整，或前後端錯誤訊息的 mapping。
  - **chat 模組**：
    - 對 `/ai-chat` / `/ai-chat/stream` 的行為調整、SSE 事件格式微調。
    - 對對話記錄／訊息 CRUD 的查詢條件和回傳形態做改良。
    - 在不破壞 handler 大設計前提下，針對某些 handler（例如 title generation 或 power deduction）做策略調整。

在這些範圍內，我可以幫你規劃比較「全面」的改動（例如一條完整鏈：前端 UI → service → controller → service/handler → DB 字段），而不只是零碎 patch。

### 3.2 暫時**不建議**直接做大規模 refactor 的區塊

有幾塊目前我只知道概念或入口，但還沒讀到能放心重構的程度：

- **整個 console 生態系統**：

  - 除了 AI Provider 以外的子系統：AiModel 管理、Secret/KeyPool、Agent 管理、Billing、RAG/Knowledge base…
  - 我大概知道頁面位置和用途，但沒有逐檔閱讀。
- **後端 Provider / Secret / Model / Agent 等模組**

  - 我只從 chat C-API 和前端 console 推斷有這些關聯，但實際的 entity/relationship、交易邏輯、邊界條件還沒看。
  - 在沒有閱讀的情況下去大改，風險會偏高（尤其跟計費、安全、權限相關）。
- **橫跨全系統的 cross‑cutting concern**

  - 完整的權限模型（除了 route.middleware 的一部分）。
  - 完整的計費與用量風控邏輯。
  - 多語系、分析/埋點、通知等。

在這些區域，如果你希望做「全面性修改」（例如整個 console permission 重構、整套 provider/secret/model 關係重設），我會建議：

1. 先為該主題開 **新的 TODO + 深入閱讀計畫**。
2. 比照 A/B/C/D 的方式，做完整一輪：前端 flow、後端 API、內部 service/handler → 產出對應 `ui-flow-*.md` 或 `api-flow-*.md`。
3. 再來設計 refactor，這樣風險更可控。
