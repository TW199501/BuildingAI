### [分析與方案]

- 分析：
  - chat 後端 C-API 的複雜度高於 A/B，因為：
    - 需要串接外部 LLM Provider，包含：
      - 模型驗證與選擇（ModelValidation）。
      - Provider 金鑰／Endpoint 取用（SecretService / ProviderService 等）。
    - 需要處理使用者用量／扣點（UserPower / PowerDeduction）。
    - 需要處理 MCP 工具呼叫與多輪工具迴圈（ToolCall、McpServer）。
    - 需要把整個多輪對話與 token 使用量落到 DB，形成 `AiChatRecord` 與 `AiChatMessage`。
  - 為了維持 ELF 風格「穩定前進」：
    - 第一輪不試圖完全理解所有 handler 的細節，只聚焦在：
      - **資料流與責任分工**：Controller → Handler → Service → Entity。
      - **關鍵欄位與錯誤處理**：對話 ID、模型 ID、訊息內容、積分／token 統計、MCP 工具錯誤等。
    - 後續若有需要再針對某一塊（例如 MCP 或扣點策略）開獨立 TODO 深入。

### [實作重點摘要]

- 已完成第一輪 chat C-API 主要程式的閱讀與整理，範圍包含：
  - `AiChatMessageWebController`（`/ai-chat`、`/ai-chat/stream`、`/ai-chat/optimize-text`）。
  - `AiChatRecordWebController`（`/ai-conversations`、`/ai-conversations/:id`、`/ai-conversations/:id/messages` 等）。
  - `AiChatRecordService`、`AiChatsMessageService`，以及一組 handler：
    - `ConversationCommandHandler`、`ChatCompletionCommandHandler`、`ModelValidationCommandHandler`、
      `UserPowerValidationCommandHandler`、`McpServerCommandHandler`、`MessageContextCommandHandler`、
      `PowerDeductionCommandHandler`、`TitleGenerationCommandHandler`、`ToolCallCommandHandler`.
- `/ai-chat`（非 streaming）的主要資料流：
  - 入口：`POST /ai-chat`，body 使用 `ChatRequestDto`，欄位包含：
    - `modelId`（必填）、`conversationId?`、`title?`、`saveConversation?`（預設為 true）、
      `messages: AiChatMessageDto[]`、`mcpServers?: string[]`.
  - 控制器流程（`AiChatMessageWebController.chat`）：
    - 在 `dto.saveConversation !== false` 時：
      - 透過 `ConversationCommandHandler.ensureConversation(user.id, conversationId, dto.title)` 建立或取得 `AiChatRecord`.
      - 取出 `dto.messages` 最後一則 user 訊息，呼叫 `saveUserMessage` 建立對應的 `AiChatMessage` 紀錄.
    - 透過 `ModelValidationCommandHandler.getAndValidateModel` 依 `modelId` 載入模型與 provider，找不到時拋出 404.
    - 透過 `UserPowerValidationCommandHandler.getAndValidateUserPower` 讀取使用者，並檢查 `power` 是否足以支應該模型的計費規則（餘額不足時直接丟出錯誤）。
    - 透過 `McpServerCommandHandler.initializeMcpServers(dto.mcpServers)`：
      - 依 MCP server ID 連線 `McpServerSSE` 或 `McpServerHttp`.
      - 取得各 MCP server 的工具清單並轉成 OpenAI `tools` 參數格式.
      - 建立 `toolToServerMap`，記錄工具名稱對應的伺服器與具體 MCP tool.
    - 透過 `MessageContextCommandHandler.limitMessageContext(dto.messages, model.maxContext)`：
      - 若存在 system 訊息，固定保留最前面一則，剩餘訊息取最後 `maxContext - 1` 個；
      - 若無 system 訊息，僅取最後 `maxContext` 個，並寫 debug log 記錄裁剪結果.
    - 呼叫 `ChatCompletionCommandHandler.executeCompletion`：
      - 使用 TextGenerator 與 provider 的 chat API 發送請求，必要時附上 `tools` 與 `tool_choice = "auto".
      - 若回應中包含 `tool_calls`：
        - 先將 assistant message push 到 messages；
        - 交給 `ToolCallCommandHandler.executeToolCalls` 逐一執行，每個結果以 `role: "tool"` 的 message 再 push 回 messages；
        - 若工具有成功執行，會在 `mcpToolCalls` 集合中累積工具呼叫紀錄，同時標記 `usedTools`.
      - 持續迴圈直到沒有更多 tool call，最終回傳 `response`、`mcpToolCalls`、`usedTools`.
    - 以 `PowerDeductionCommandHandler.calculateConsumedPower(total_tokens, billingRule)` 將 token 使用量依 `billingRule` 轉成實際扣點金額，並透過 `AppBillingService.deductUserPower` 落帳；若帳務過程失敗，只記錄 log、不丟出錯誤，以避免影響已完成的聊天結果.
    - 若 `saveConversation !== false` 且有 `conversationId` 與最終 AI 回覆：
      - `ConversationCommandHandler.saveAssistantMessage` 會將 AI 回覆內容、token usage、MCP 工具呼叫紀錄（如有）寫入 `AiChatMessage`.
      - `TitleGenerationCommandHandler.generateTitle` 會再額外呼叫一次 LLM 產生短標題，並在失敗時退回 user 訊息前 20 字作為 fallback.
    - 若存在 `total_tokens` 且模型有計費規則：
      - 呼叫 `PowerDeductionCommandHandler.deductUserPower`，以交易方式請 `AppBillingService` 實際扣除使用者積分並寫入帳務紀錄；若扣點過程發生錯誤，僅記錄 log，不會讓這次聊天失敗.
    - 組合回應物件：`{ ...finalResponse, conversationId, mcpInfo }`，其中 `mcpInfo` 只列出實際使用過的 MCP server 與工具摘要，方便前端展示.
- `/ai-chat/stream`（streaming SSE）的主要資料流：
  - 入口：`POST /ai-chat/stream`，同樣使用 `ChatRequestDto`，回應改為 SSE；由 `AiChatMessageWebController.chatStream` 負責.
  - 差異重點：
    - 一開始即呼叫 `getAndValidateModel` 與 `getAndValidateUserPower`，提早在真正 streaming 前做模型與餘額檢查.
    - 在 `saveConversation !== false` 時先 `ensureConversation` 並保存 user 訊息，接著立刻透過 SSE 傳回：
      - `data: { type: "conversation_id", data: <conversationId> }`，讓前端可以及早知道會話 ID.
    - 初始化 MCP server 與 tools 後，呼叫 `ChatCompletionCommandHandler.executeStreamCompletion`，該 handler：
      - 以 streaming 方式呼叫 provider，並在每個 chunk 中：
        - 若有文字內容，輸出 `type: "chunk"` 事件並累積到 `fullResponse`.
        - 若有 reasoning content，輸出 `type: "reasoning"` 事件，並記錄 reasoning 的內容與開始／結束時間.
        - 若偵測到 tool_calls，會先輸出 `type: "mcp_tool_detected"`，後續實際執行時再輸出 `mcp_tool_start` / `mcp_tool_result` / `mcp_tool_error` 等事件.
      - 每一輪 tool call 結束後，將工具輸出以 `role: "tool"` 的訊息加入 messages，必要時再進入下一輪 AI 呼叫，直到不再有 tool call.
      - 最終回傳 `fullResponse`（完整 AI 回覆字串）、`finalChatCompletion`、`mcpToolCalls`、`reasoningContent` 與推理時間資訊.
    - streaming 結束後，若 `saveConversation !== false` 且有 `conversationId` 且 `fullResponse` 非空：
      - 以 `saveAssistantMessage` 將 AI 回覆寫入 DB，並在 metadata 中記錄 reasoning 內容與耗時（若有）。
      - 若有 `reasoningContent` 則改用 `generateTitleFromReasoning` 產生標題，否則仍使用 `generateTitle`.
    - 同樣以 `calculateConsumedPower` + `deductUserPower` 完成計費與扣點，然後透過 `cleanupMcpServers` 關閉 MCP 連線，最後輸出 `data: [DONE]` 結束 SSE.
    - 例外處理：
      - 任一階段出錯時會先清理 MCP 連線；若有 `conversationId`，會以 `saveAssistantMessage` 寫入一則帶有 `errorMessage` 的訊息（content 留空、token 為 0）。
      - 之後透過 SSE 傳回 `type: "error"` 的 JSON，最後同樣輸出 `[DONE]` 並關閉連線；若在寫 SSE 過程本身也出錯，則記錄 log 並丟出 `HttpErrorFactory.badRequest`.
- `/ai-conversations` 相關 API 的主要資料流：
  - 入口控制器：`AiChatRecordWebController`，透過 `AiChatRecordService` / `AiChatsMessageService` 執行實際 CRUD.
  - `GET /ai-conversations`：
    - 使用 `QueryAIChatRecordDto` 作為查詢條件，支援 `modelId`、`status`、`isPinned`、`keyword`、`username` 等欄位，並固定排除 `isDeleted = true` 的紀錄.
    - 內部使用 `findUserConversations(userId, queryDto, includeUserInfo=false)`，排序規則為 `isPinned DESC`、`updatedAt DESC`，對應前端會話列表的顯示行為.
  - `GET /ai-conversations/:id`：
    - 透過 `getConversationWithMessages(conversationId, user.id)` 取得單一對話，並限制只能讀取自己的紀錄；若對話不存在或不屬於當前使用者，會拋出對應錯誤.
  - `PATCH /ai-conversations/:id`：
    - 在 `updateConversation` 中先用 where 條件檢查記錄是否存在、是否被刪除、是否屬於當前 user，僅在通過檢查時才允許更新（例如修改標題、置頂狀態等）。
  - `DELETE /ai-conversations/:id` 與批次刪除：
    - 採 soft delete 方式，將 `isDeleted` 設為 true，並依 userId 決定是否為管理員操作；刪除失敗時會記錄錯誤並拋出對應的 HttpErrorFactory.
  - `PATCH /ai-conversations/:id/pin`：
    - 透過 `pinConversation` 更新 `isPinned` 欄位，同樣會檢查對話是否存在與權限是否正確.
  - `GET /ai-conversations/:id/messages`：
    - 先以 `getConversationWithMessages` 檢查對話是否屬於當前 user，若不存在則直接丟出錯誤.
    - 再呼叫 `getConversationMessages(conversationId, paginationDto)`，下層由 `AiChatsMessageService` 以 `sequence DESC` 排序並分頁；對應前端 infinite scroll 在載入更多歷史訊息時的行為（前端會再反轉順序顯示）。
- handler / DTO 責任分工小結（供之後查表）：
  - `ChatRequestDto`：定義 chat 請求的欄位與驗證，`saveConversation` 透過 `isEnabled` 轉 boolean，預設為 true，並允許附帶 MCP server ID 陣列.
  - `ModelValidationCommandHandler`：只負責載入模型與對應 provider，找不到模型就直接丟出 notFound.
  - `UserPowerValidationCommandHandler`：負責確認使用者存在且 power 大於 0（在該模型有計費規則時），不足時回傳「餘額不足」類型錯誤.
  - `McpServerCommandHandler` + `ToolCallCommandHandler`：
    - 前者管理 MCP server 的連線／斷線與工具清單轉換，允許部份 MCP 連線失敗但不影響整體聊天流程（只會寫 warning log）。
    - 後者統一執行工具並包裝成 `McpToolCall` 結構（含 input/output/duration/status 等），同時回傳給 ChatCompletion handler 用於後續儲存與 SSE 回傳.
  - `MessageContextCommandHandler`：確保不會把超過 `maxContext` 的 messages 丟給 LLM，並在 log 中記錄裁剪前後的訊息數量，方便除錯.
  - `PowerDeductionCommandHandler`：將 token 使用量依 `billingRule` 轉成實際扣點金額，並透過 `AppBillingService.deductUserPower` 落帳；若帳務過程失敗，只記錄 log、不丟出錯誤，以避免影響已完成的聊天結果.
  - `TitleGenerationCommandHandler`：在對話完成後額外呼叫一次 LLM 產生短標題，並在失敗時退回 user 訊息前 20 字；對 reasoning model 會改用 `generateTitleFromReasoning` 直接從 user 訊息截斷.
  - `ConversationCommandHandler`：封裝「建立／查詢／更新對話與訊息」這一層，並將 `CHAT_ROLE` 映射為 `MessageRole`、統一處理 user/assistant 訊息的落 DB 操作.
